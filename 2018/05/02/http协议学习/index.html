http协议简介

http协议（HyperText Transfer Protocol，超文本传输协议）是一个属于应用层的面相对象的协议。Ted Nelson组织协调万维网协会（World Wide Web Consortium）和互联网工程工作小组（Internet Engineering Task Force ）共同合作研究，最终发布了一系列的RFC，其中著名的RFC 2616定义了HTTP 1.1。

http协议的特点

  • 支持C/S模式
  • 简单快速：客户向服务器请求服务时，只需传送请求方法和路径。常用的请求方法有GET、POST、HEAD。
  • 灵活：http协议允许传送任意类型的数据对象，正在传输的类型由content-type加以标记。
  • 无连接：无连接的含义是限制每次连接只处理一个请求，服务器处理网客户请求，收到客户应答后就断开连接。采用这种方式，可以节省传输时间。
  • 无状态：无状态是指对于事物处理没有记忆能力，就是说后续任务需要前面任务的信息的话就要重新传送，这样可能导致每次连接的传输量增大，不过，在不需要先前信息时应答就会很快。

URL详解

http协议使用统一资源标识符（Uniform Resource Identifier，URI）来传输数据和建立连接。URL（Uniform Resource Locator，统一资源定位符）是一种特殊类型的URI，它包含了用于查找某个资源的足够多的信息。

URL的组成部分

示例：http://www.aspxfans.com:8080/news/index.asp?boardID=5&ID=24618&page=1#name

一个完整的URL包括以下部分：

  • 协议部分：该URL的协议部分是“http:”，后面的“//”是分割符。
  • 域名部分：该URL的域名部分是“www.aspxfans.com”，也可用ip地址表示，如“120.77.180.118”。
  • 端口部分：跟在域名后面的是端口，用“:”分隔，该URL的端口部分是8080，端口部分不是必须的部分，如果省略的话，将采用默认端口。
  • 虚拟目录部分：从域名后的第一个“/”到最后一个“/”是URL的虚拟目录部分，该URL的虚拟目录部分是“/news/”。
  • 文件名部分：从最后一个“/”到“?”之间是文件名部分，如果没有“?”,则是从最后一个“/”到“#”为止，如果没有“#”，则从最后一个“/”到最后，该URL的文件名部分是“index.asp”。文件名部分也不是必须的部分，省略的话，将采用默认文件名。
  • 参数部分：从“?”到“#”之间的部分是参数部分，可以允许有多个参数，用“&”分隔。该URL的参数部分为“boardID=5&ID=24618&page=1”。
  • 锚部分：从“#”到最后是锚部分，该URL的锚部分是“name”。锚部分也不是必须的部分。

http请求详解

http request

一个http请求由请求行（request line）、请求头（header）、空行和请求数据四部分组成。

  1. 第一部分：请求行

    用来说明请求类型，要访问的资源以及协议版本。

  2. 第二部分：请求头

    用来说明服务器要使用的附加信息。

  3. 第三部分：空行

    请求头后的空行是必须的，没有请求数据也要有这个空行。

  4. 第四部分：请求数据

    请求数据也叫主体，请求数据可以没有。

  • 请求示例

    • 第一部分：请求行（第1行），表明了请求方式-POST，资源文件-jpg，协议及版本
    • 第二部分：请求头（第2-6行），表明一些请求信息
    • 第三部分：空行（第7行）
    • 第四部分：请求主体（第8行及以后）

http response

一般情况下，服务器收到http请求后会返回一个http响应（response）。http响应也由四个部分组成–状态行、消息报头、空行和响应正文。

  • http 状态码
    状态代码有三位数字组成，第一个数字定义了响应的类别，共分五种类别:
    1xx：指示信息–表示请求已接收，继续处理
    2xx：成功–表示请求已被成功接收、理解、接受
    3xx：重定向–要完成请求必须进行更进一步的操作
    4xx：客户端错误–请求有语法错误或请求无法实现
    5xx：服务器端错误–服务器未能实现合法的请求

  • 常用状态码：

    | 状态码 | 英文描述 | 中文说明 |
    | — | — | — |
    | 200 | OK | 请求成功 |
    | 301 | 永久移动 | 永久移动。请求的资源已被永久的移动到新URI，返回信息会包括新的URI，浏览器会自动定向到新URI。今后任何新的请求都应使用新的URI代替 |
    | 400 | Bad Request | 客户端请求的语法错误，服务器无法理解 |
    | 404 | Not Found | 服务器无法根据客户端的请求找到资源 |
    | 500 | Internal Server Error | 服务器内部错误，无法完成请求 |
    | 502 | Bad Gateway | 充当网关或代理的服务器，从远端服务器接收到了一个无效的请求 |

    状态码详细说明请移步http://www.runoob.com/http/http-status-codes.html

POST与GET的区别

  1. GET请求，请求的参数会附加在URL后面，以“?“分隔，多个参数之间用”&“连接。因此，GET的参数会暴露在URL中，而POST的参数隐藏在请求主体中。相对而言POST更安全一些。
  2. GET在浏览器回退时是无害的，而POST请求会请求再次提交
  3. 对于参数的类型，GET只接受ASCII类型，POST无限制。
  4. GET请求在URL中传送的参数是有长度限制的，而POST没有。大多数浏览器URL长度限定在2kB、服务器限定在64kB。

但是GET与POST方式本质上是没有区别的，POST和GET都是TCP连接，在POST方式的URL后加上参数，在GET请求体里加上参数是能够做到的，但是一般不允许这么做，http协议是个行为准则，但还是基于TCP协议实现的。

两者另一个重大区别是GET产生一个TCP数据包，POST产生两个TCP数据包。POST需要先发送header，服务器响应100后，在发送data，服务器响应200。而GET只有一次发送，得到相应结果。也不是说GET的效率就一定比POST好，在网络环境好的时候，发一次包和发两次包的时间差别可以忽略，而在网络环境不好的时候，POST在验证数据完整性上有很大的优点。

当你在浏览器地址栏输入了facebook.com回车后，发生了什么？

  1. 转换非ASCII字符

    浏览器检查输入是否含有不是a-z，A-Z，0-9，- 或者.的字符，如果有，将这些字符进行转码。

  2. 检查HSTS列表

    浏览器检查自带的“预加载 HSTS（HTTP严格传输安全）”列表，这个列表里包含了那些请求浏览器只使用HTTPS进行连接的网站。如果要访问的网站在这个列表里，浏览器会使用HTTPS协议，而非HTTP协议。

  3. DNS查询

    • 查询浏览器缓存：浏览器会记录DNS一段时间，在Chrome中可用在chrome://net-internals/#dns查看。
    • 查询系统缓存：如果浏览器缓存没有相应记录，就会发起系统调用查看系统缓存
    • 查询路由器缓存：接着前面的请求会去查询路由器，路由器有自己的DNS缓存。

    • 查询本地域名服务器，若有相关缓存，就会告知主机，若没有，则会向更高级的域名服务器查询。

      • 本机向本地域名服务器的查询一般采用递归查询（recursive query）。递归查询就是：如果本机所询问的本地域名服务器不知道被查询域名的IP地址，那么本地域名服务器就以DNS客户的身份，向其他根域名服务器发出查询请求报文（代替该主机继续查询），而不是让该主机自己进行下一步查询。因此，递归查询返回的查询结果或者是所要查询的IP地址，或者是报错，或者是无法查询到所需的IP地址。

      • 本地域名服务器向根域名服务器的查询通常是采用迭代查询（iterative query）。迭代查询是当根域名服务器收到本地域名服务器发出的查询请求报文时，要么给出要查询的IP地址，要么告知下一步要查询的域名服务器，而不会代替主机进行查询。当然查询方式是可以在请求报文中设置的。

  4. 浏览器给web服务器发送一个HTTP请求

    GET 这个请求定义了要读取的URL： “http://facebook.com/”。 浏览器自身定义 (User-Agent 头)， 和它希望接受什么类型的相应 (Accept andAccept-Encoding 头). Connection头要求服务器为了后边的请求不要关闭TCP连接。

    请求中也包含浏览器存储的该域名的cookies。可能你已经知道，在不同页面请求当中，cookies是与跟踪一个网站状态相匹配的键值。这样cookies会存储登录用户名，服务器分配的密码和一些用户设置等。Cookies会以文本文档形式存储在客户机里，每次请求时发送给服务器。

  5. 服务器永久重定向301响应

    服务器给浏览器响应一个301永久重定向响应，这样浏览器就会访问“http://www.facebook.com/” 而非“http://facebook.com/” 。如果同时将几个域名解析到同一个空间的时候，如果不做重定向，就会形成重复页面，对网站是极度不利的，这种情况需要做301重定向。

  6. 浏览器跟踪重定向

    现在，浏览器知道了“http://www.facebook.com/” 才是要访问的正确地址，所以它会发送另一个获取请求：

  1. 服务器处理请求
  2. 服务器返回一个响应
  3. 浏览器渲染并显示
